## Overview

All runtime configuration is via **environment variables** stored in a `.env` file at the project root. This follows the [12-factor app methodology](https://12factor.net/config) — configuration varies between deployments (dev, staging, prod) but code does not.

**Why environment variables?**
- Secrets never appear in source code
- Docker Compose passes them to all containers automatically
- Can be overridden per-environment without changing files
- Standard pattern understood by all deployment tools (Docker, K8s, CI/CD)

## Environment Variables

| Variable | Required | Default | Description | Why This Default |
|----------|----------|---------|-------------|-----------------|
| `POSTGRES_DB` | Yes | — | Database name | No default — forces explicit naming |
| `POSTGRES_USER` | Yes | — | Database user | No default — avoids `postgres` superuser |
| `DB_PASSWORD` | **Yes** | — | Database password | Auto-generated by `setup-env.sh` (32 chars) for security |
| `JWT_SECRET` | **Yes** | — | JWT signing secret | Must be 64+ chars. Auto-generated. Short secrets are brute-forceable |
| `JWT_EXPIRY` | No | `24h` | Access token lifetime | 24h balances security (limits exposure of stolen token) with UX (no mid-day re-login) |
| `REFRESH_TOKEN_EXPIRY` | No | `7d` | Refresh token lifetime | 7 days covers a work week without re-authentication |
| `PORT` | No | `4000` | Backend listen port | 4000 avoids conflict with common services (3000=React, 5432=PG, 8080=other) |
| `NODE_ENV` | No | `production` | Environment mode | Defaults to `production` safety — dev features must be opted-in, not opted-out |
| `LOG_LEVEL` | No | `info` | Winston log level | `info` captures HTTP requests and CRUD operations without debug noise |
| `ADMIN_EMAIL` | No | `admin@esg.local` | Initial admin email | `.local` TLD signals it should be changed for real deployments |
| `ADMIN_PASSWORD` | **Yes** | — | Initial admin password | No default — forces a secure password choice |
| `ADMIN_NAME` | No | `System Administrator` | Initial admin name | Descriptive placeholder |

### How Variables Flow Through the System

```
.env file
  ↓ (docker compose reads it)
docker-compose.yml → environment:
  ↓
Container process.env
  ↓
backend/src/config/env.ts → exports { env } object
  ↓
Used by: app.ts, auth.service.ts, logger.ts, database.ts
```

The `env.ts` module centralizes all environment access:

```typescript
// backend/src/config/env.ts
export const env = {
  PORT: parseInt(process.env.PORT || '4000'),
  NODE_ENV: process.env.NODE_ENV || 'production',
  JWT_SECRET: process.env.JWT_SECRET!,  // Throws if missing
  // ... all other vars
};
```

**Why centralize?** A single module prevents scattered `process.env` access across the codebase, makes required variables explicit, applies defaults consistently, and makes testing easier (mock one module, not every `process.env` usage).

## Initial Setup

### Using the Setup Script (Recommended)

```bash
./scripts/setup-env.sh
```

The script:
1. Prompts for admin email and password
2. Generates a **cryptographically random** `DB_PASSWORD` (32 chars via `openssl rand`)
3. Generates a **cryptographically random** `JWT_SECRET` (64 chars via `openssl rand`)
4. Creates `.env` from the `.env.example` template
5. Sets restrictive file permissions (`chmod 600 .env`)

**Why a setup script?** Manual `.env` creation leads to:
- Weak passwords (`password123`)
- JWT secrets that are too short (< 32 chars)
- Forgotten variables → runtime crashes
- No consistent file permissions

### Manual Setup

```bash
cp .env.example .env
# Edit .env and fill in required values:
#   DB_PASSWORD  — use: openssl rand -base64 32
#   JWT_SECRET   — use: openssl rand -base64 64
#   ADMIN_PASSWORD — minimum 8 characters
```

### Verifying Configuration

```bash
# Check .env exists and isn't tracked
cat .env                    # Should show all variables filled
git status .env             # Should NOT appear (in .gitignore)

# Check Docker sees the variables
docker compose config       # Shows interpolated docker-compose.yml

# Check backend loaded them
docker compose exec backend env | grep -E "PORT|NODE_ENV|LOG_LEVEL"
```

## Configuration Files

| File | Purpose | Why This Format |
|------|---------|----------------|
| `.env` | Runtime secrets & config | Standard for Docker Compose; never committed |
| `.env.example` | Template with all variables | Documents required vars; safe to commit (no secrets) |
| `package.json` (root) | Root workspace scripts | Cleanup, API spec validation, and setup commands |
| `redocly.yaml` | API spec lint rules | Configures Redocly CLI for OpenAPI validation |
| `api-specs/openapi.yaml` | OpenAPI 3.0.3 specification | Machine-readable API contract (version: 2026-02-24) |
| `backend/tsconfig.json` | TypeScript compiler options | Strict mode enabled for type safety |
| `backend/jest.config.js` | Jest test runner config | JS (not JSON) for comments and conditional logic |
| `backend/prisma/schema.prisma` | Database schema | Prisma's declarative DSL; generates types + migrations |
| `frontend/vite.config.ts` | Vite build tool config | TypeScript for type-checked config |
| `frontend/tailwind.config.js` | Tailwind CSS config | Configures design tokens (colors, spacing) |
| `frontend/postcss.config.js` | PostCSS plugins | Required by Tailwind for CSS processing |
| `docker-compose.yml` | Container orchestration | Declarative service definitions; single `up` command |
| `frontend/nginx.conf` | Nginx web server | Reverse proxy config + SPA routing |
| `.gitleaks.toml` | Credential scanner rules | Defines secret patterns + allowlists |
| `scripts/cleanup.sh` | Cleanup wizard | Docker, build, log, and test artifact cleanup |

### Key Configuration Choices Explained

**TypeScript Strict Mode** (`"strict": true` in `tsconfig.json`)
- Enables `strictNullChecks`, `noImplicitAny`, `strictFunctionTypes`
- Catches entire classes of bugs at compile time (null derefs, implicit any, wrong function signatures)
- Worth the extra type annotations for a data platform where correctness matters

**Jest `clearMocks: true` + `restoreMocks: true`** (in `jest.config.js`)
- `clearMocks` — resets call counts between tests (prevents test interdependence)
- `restoreMocks` — restores original implementations after each test (prevents mock leaks)
- Together they enforce test isolation without boilerplate `afterEach` blocks

**Prisma `schema.prisma` as single source of truth**
- Database tables, indexes, relations, and TypeScript types all derive from one file
- `prisma migrate dev` generates SQL migrations automatically
- `prisma generate` creates the typed client — no manual ORM model classes

## Logging Configuration

### Why Winston + Morgan?

**Winston** handles structured logging (JSON in production, colorized in dev) with file rotation. **Morgan** is Express HTTP request logging middleware. Morgan pipes through Winston so all logs (application + HTTP) flow through a single channel with consistent formatting.

| Config | Value | Why |
|--------|-------|-----|
| Console output | Colorized (dev), JSON (prod) | Human-readable in dev; machine-parsable for log aggregators (ELK, Datadog) in prod |
| Error log file | `logs/error.log` | Separate error file for quick triage — `tail -f logs/error.log` shows only errors |
| Combined log file | `logs/combined.log` | Full audit trail of all operations |
| File rotation | 10 MB max, 5 files retained | Prevents disk exhaustion; 50 MB max per log type |
| Docker log rotation | JSON driver, 10 MB, 5 files | Same rationale at the container level — prevents `docker logs` from consuming disk |
| HTTP request logging | Morgan `combined` format | Includes IP, method, URL, status, response time, user agent — standard for web servers |

### Log Levels

| Level | When Used | Examples |
|-------|-----------|---------|
| `error` | Unrecoverable failures | Database connection lost, unhandled exceptions, Prisma query failures |
| `warn` | Recoverable issues | Failed login attempts, validation rejection, rate limit hit |
| `info` | Normal operations | User login, CRUD operations, HTTP requests, server start |
| `debug` | Diagnostic detail | SQL queries, JWT decode details, request/response bodies |

**How to change**: Set `LOG_LEVEL=debug` in `.env` and restart. In Docker: `docker compose down && docker compose up -d`.

## Docker Compose Configuration

### Service Architecture

| Service | Image | Internal Port | External Port | Depends On | Why |
|---------|-------|---------------|---------------|------------|-----|
| `frontend` | Custom (Nginx + React build) | 80 | **3000** | backend | Nginx reverse-proxies `/api/*` to backend; serves React SPA for everything else |
| `backend` | Custom (Node.js 20) | 4000 | **4000** | db | Express API server; waits for DB to be healthy before starting |
| `db` | postgres:16-alpine | 5432 | **5432** | — | PostgreSQL with alpine image for smaller footprint (< 80 MB) |

### Volume Persistence

```yaml
volumes:
  postgres_data:    # Named volume — PostgreSQL data survives container restarts
```

**Why a named volume?** Bind mounts (`./data:/var/lib/postgresql/data`) have permission issues on some OSes. Named volumes are managed by Docker and work consistently across Linux/macOS/Windows.

### Network

All 3 services share a default Docker Compose network. Services reference each other by container name:
- Backend connects to DB via `DATABASE_URL=postgresql://user:pass@db:5432/dbname`
- Nginx proxies to backend via `proxy_pass http://backend:4000`

### Health Checks

```yaml
db:
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U $POSTGRES_USER"]
    interval: 5s
    timeout: 5s
    retries: 5
```

**Why?** The backend container must wait for PostgreSQL to accept connections before running Prisma migrations. Without a health check, the backend would crash on startup if the DB isn't ready.

## Rate Limiting

| Endpoint | Window | Max Requests | Why |
|----------|--------|-------------|-----|
| `POST /api/2026-02-24/auth/login` | 15 min | 10 | Prevents brute-force password attacks. 10 attempts covers typos without enabling credential stuffing. |
| All `/api/2026-02-24/*` | 1 min | 200 | Prevents DoS from a single client. 200/min is generous for human users but stops automated abuse. |

**Implementation**: `express-rate-limit` middleware applied in `app.ts`. Rate limits are per-IP. Behind a reverse proxy (Nginx), set `app.set('trust proxy', 1)` to use `X-Forwarded-For`.

## Development vs Production

| Setting | Development | Production |
|---------|------------|------------|
| `NODE_ENV` | `development` | `production` |
| Console log format | Colorized, human-readable | JSON (machine-parsable) |
| Log level | `debug` (recommended) | `info` |
| Error stack traces | Included in API responses | Excluded (generic "Internal server error") |
| Prisma Studio | Available (`npx prisma studio`) | Not exposed |
| Hot reload | `npm run dev` (tsx watch) | `npm start` (compiled JS) |
| Docker | Optional (can run directly) | Required (standardized environment) |
| Source maps | Inline | Disabled |

### Running in Development Mode

```bash
# Without Docker — direct Node.js
cd backend
export NODE_ENV=development
export JWT_SECRET=dev-secret-change-me
export DATABASE_URL="postgresql://user:pass@localhost:5432/esg_dev"
npm run dev    # tsx watch mode — hot reload on file changes

# With Docker — still uses production mode by default
# Override by adding NODE_ENV=development in docker-compose.yml
```

## Changing Configuration

### Adding a New Environment Variable

1. **Add to `.env.example`** with a comment explaining its purpose
2. **Add to `backend/src/config/env.ts`** with default or throw-if-missing logic
3. **Add to `docker-compose.yml`** under `environment:` for the relevant service
4. **Update this document** with the variable details
5. **Update `setup-env.sh`** if it needs auto-generation or prompting

### Changing a Default

1. Update the default in `env.ts`
2. Update this document
3. Rebuild: `docker compose build backend && docker compose up -d`

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| Backend crashes on start | Missing `JWT_SECRET` | Run `./scripts/setup-env.sh` or add to `.env` |
| DB connection refused | DB container not healthy | `docker compose ps` — check db health status |
| `.env` changes not picked up | Docker cached old env | `docker compose down && docker compose up -d` |
| Port 4000 already in use | Another process on port | `lsof -i :4000` to find and kill it, or change `PORT` |
| Prisma migration fails | Schema mismatch | `docker compose exec backend npx prisma migrate reset` (⚠️ deletes data) |
| Logs not appearing | `LOG_LEVEL` too high | Set `LOG_LEVEL=debug` in `.env` and restart |
| Rate limit hit in tests | Shared IP across test requests | Tests use mocked routes; rate limiter doesn't apply in test mode |
